## 1. CISC (Complex Instruction Set Computer)

- 개념: "복잡하고 많은 명령어 세트를 가진 컴퓨터"
- 철학: 하드웨어(CPU)가 똑똑해야 한다. 명령어 한 줄로 복잡한 작업을 끝내자!
- 동작 방식: 명령어 하나의 길이가 가변적이며, 하나의 명령어가 여러 단계의 복잡한 연산을 수행합니다.
- 장점: 컴파일러가 만들어내는 코드의 양(프로그램 크기)이 줄어들어 메모리를 아낄 수 있습니다.
- 단점: CPU 내부 회로가 매우 복잡해지고, 명령어 하나를 실행하는 데 걸리는 시간(클럭 사이클)이 제각각입니다.
- 대표 주자: Intel, AMD (x86 아키텍처) → 우리가 쓰는 대부분의 데스크톱, 노트북 PC
## 2. RISC (Reduced Instruction Set Computer)

- 개념: "단순하고 감소된 명령어 세트를 가진 컴퓨터"
- 철학: 명령어를 최대한 단순하게 만들자. 복잡한 건 소프트웨어(컴파일러)가 조합하면 된다!
- 동작 방식: 명령어의 길이를 고정하고 종류를 줄였습니다. 모든 명령어가 규칙적이고 단순하게 실행됩니다.
- 장점: CPU 구조가 단순해서 전력 소모가 적고, 명령어 하나가 1클럭 내외로 빠르게 처리되어 효율적입니다. (파이프라이닝 최적화 유리)
- 단점: 단순한 명령어를 조합해야 하므로 프로그램 코드의 전체적인 크기가 CISC에 비해 커집니다.
- 대표 주자: ARM, Apple Silicon (M1/M2/M3 등) → 스마트폰, 태블릿, 맥북, 임베디드 장치
## 요약 표

| 구분 | CISC (컴퓨터가 복잡함) | RISC (명령어가 단순함) |
|---|---|---|
| 명령어 수 | 많고 복잡함 (가변 길이) | 적고 단순함 (고정 길이) |
| 연산 방식 | 명령어 하나가 여러 연산 수행 | 단순 명령어 여러 개를 조합하여 수행 |
| 전력 소모 | 비교적 높음 (발열 많음) | 낮음 (효율적인 전력 소모) |
| 주요 장점 | 코드 크기가 작음, 하드웨어 중심 | 속도가 빠름, 전력 효율 좋음, 소프트웨어 중심 |
| 주요 제품 | x86 (Intel Core, AMD Ryzen 등) | ARM (갤럭시/아이폰 AP, Apple M 시리즈) |

> 💡 최신 트렌드 한마디: > 최근에는 CISC CPU(Intel) 내부적으로는 RISC 형태로 변환해 실행하고, RISC CPU(ARM)도 성능 향상을 위해 복잡한 기능을 추가하면서 두 방식의 경계가 많이 모호해지고 융합되는 추세입니다.

C언어 코드를 컴파일할 때, 컴파일러는 본인의 컴퓨터 CPU 아키텍처(CISC 또는 RISC)에 맞는 기계어 코드로 변환해 주게 됩니다.



Unicode(유니코드)와 UTF-8을 사용하는 이유는 한마디로 "전 세계 모든 나라의 언어를 깨짐 없이 하나의 표준으로 표현하기 위해서"입니다.

이 둘이 나오기 전에는 국가마다, 컴퓨터마다 문자 인코딩 방식(국내의 경우 EUC-KR 등)이 달라서 해외 사이트에 접속하거나 다른 언어로 된 파일을 열면 글자가 ``나 뷁뷄처럼 깨지는 현상이 빈번했습니다.

## 1. Unicode (유니코드) 란?

- 정의: 전 세계의 모든 문자(한글, 영어, 한자, 아랍어, 이모지 등)에 고유한 숫자(코드 포인트)를 부여한 일종의 '문자 지도(사전)'입니다.
- 사용하는 이유: 과거에는 영어권의 ASCII(7비트)나 한국의 EUC-KR처럼 지역별로 다른 표준을 썼기 때문에, 한 문서에 여러 나라 언어를 쓰면 글자가 깨졌습니다. 이를 해결하기 위해 "전 세계 문자를 단 하나의 표준 숫자로 통일하자!"고 만든 것입니다.
- 예시: * 문자 'A' → 유니코드 주소: U+0041
## 2. UTF-8 이란?

- 정의: 유니코드 사전에 적힌 숫자를 컴퓨터(메모리)에 실제로 저장하는 방식(인코딩 방식) 중 하나입니다. 유니코드를 표현하는 가장 인기 있는 방식입니다.
- 사용하는 이유 (핵심 기술: 가변 길이 인코딩):
### 💡 왜 이 둘을 함께 써야 할까요? (차이점 직관 이해)

두 개념은 개념적 역할이 완전히 다릅니다.

- Unicode: "무슨 문자가 어떤 숫자인지 매핑해 둔 글자 사전"
- UTF-8: "그 사전에 있는 숫자를 메모리에 효율적으로 저장하는 방법"
### 요약 표

| 구분 | Unicode (유니코드) | UTF-8 |
|---|---|---|
| 역할 | 문자 집합 (Character Set) | 인코딩 방식 (Encoding) |
| 핵심 목적 | 전 세계 문자의 고유 숫자 번호판 표준화 | 유니코드 숫자를 메모리에 효율적으로 저장 |
| 크기 | 논리적인 코드 값 (주소 개념) | 문자에 따라 1~4바이트 가변 크기 |
| 가장 큰 장점 | 다국어 지원 (글자 깨짐 문제 완전 해결) | 기존 ASCII(영어)와 완벽 호환, 메모리 절약 |

# 📌 UTF-8 3바이트 틀 (한글 저장 규칙)

### 1. 3바이트 틀 개요

유니코드 주소가 0x0800 ~ 0xFFFF 사이에 있는 문자(한글, 한자 등)는 UTF-8에서 무조건 아래의 고정된 틀을 사용합니다.

1110xxxx  10xxxxxx 10xxxxxx

- 1110 / 10 / 10: 컴퓨터가 문자 경계를 인식하기 위한 국제 표준 고정 표식
- xxxx...: 실제 유니코드 비트가 쪼개져 들어가는 데이터 공간 (총 16비트)
### 2. 고정 표식의 역할 (컴퓨터가 해석하는 법)

컴퓨터는 데이터(0과 1)를 1바이트(8비트)씩 읽으면서 첫머리 비트를 보고 글자 크기를 판단합니다.

- 0xxxxxxx (1바이트 구역): "첫 비트가 0이네? 영어/숫자구나. 1바이트만 읽어야지."
- 110xxxxx (2바이트 구역): "첫머리가 110이네? 다음 바이트까지 총 2바이트가 한 글자구나."
- 1110xxxx (3바이트 구역): "첫머리가 1110이네? 다다음 바이트까지 총 3바이트가 한 글자(한글)구나."
- 10xxxxxx (이어지는 바이트): "앞이 10이네? 나는 혼자서는 의미가 없고, 앞 문자에 종속된 데이터구나."
### 3. 핵심 요약 (노션 한 줄 정리)

> Q. 이 틀은 무조건 고정인가요?

[유니코드]
김: AE40 (1010 1110 0100 0000)
호: D638 (1101 0110 0011 1000)
철: CCA0 (1100 1100 1010 0000)

[UTF-8]
김: 1110[1010] 10[111001] 10[000000]
호: 1110[1101] 10[011000] 10[111000]
철: 1110[1100] 10[110010] 10[100000]



```c
#define _CRT_SECURE_NO_WARNINGS // Visual Studio에서 scanf 등 구형 함수의 경고를 끄기 위한 설정

#include <stdio.h>  // 표준 입출력 라이브러리 (printf, scanf, fgets 등)
#include <string.h> // 문자열 처리 라이브러리 (strlen, strcpy, strcmp 등)
#include <stdlib.h> // 표준 유틸리티 라이브러리 (atoi, malloc, free 등)
```

```c
int main(void)
{
	char buff[10];
	int num1, num2;
	char operator;
	int result = 0;

	while (1)
	{
		// 1. 명령 및 안내 메시지 출력
		printf("사칙연산(1+2=) 종료: quit > ");

		// 2. 키보드 입력 받기 (stdin)
		// [quit] 입력 시 buff 저장 방식:
		// index :  0   1   2   3   4   5
		// value : 'q' 'u' 'i' 't' '\n' '\0'
		fgets(buff, sizeof(buff), stdin);

		// 3. 종료 조건 검사
		if (buff[0] == 'q' && buff[1] == 'u' && buff[2] == 'i' && buff[3] == 't')
		{
			printf("프로그램을 종료합니다.\n");
			break;
		}

		// 4. 데이터 변환 (ASCII 문자 -> 정수 int)
		num1 = buff[0] - '0'; // buff[0] - 0x30; 과 동일
		operator = buff[1];
		num2 = buff[2] - '0';

		// 5. 사칙연산 계산
		switch (operator)
		{
		case '+':
			result = num1 + num2;
			break;
		}

		// 6. 결과 출력 (정수 -> ASCII 문자 변환 후 출력)
		printf("%c + %c = %c\n", num1 + '0', num2 + '0', result + '0');
	}

	return 0;
}
```



## 📌 C언어 핵심 문자열 함수 요약 (strncmp, atoi, itoa)

### 1. strncmp : 문자열 n개 비교

- 기능: 두 문자열을 지정한 개수(n)만큼만 앞에서부터 비교합니다.
- 헤더: <string.h>
- 핵심 활용: 뒤에 엔터(\n)나 쓰레기 값이 붙어 있는 배열에서 앞부분의 특정 키워드(예: "quit")만 정확히 골라내어 비교할 때 안전하게 사용합니다.
```c
strncmp(str1, str2, n);
// 결과가 0이면 -> 지정한 n개 문자열이 완벽히 일치함
```

### 2. atoi : 문자열 → 정수 변환 (Ascii To Integer)

- 기능: 숫자로 된 문자열을 진짜 계산 가능한 정수(int)로 변환합니다.
- 헤더: <stdlib.h>
- 핵심 활용: fgets() 등으로 입력받은 문자 배열 "123"을 숫자 123으로 바꾸어 사칙연산을 하고 싶을 때 사용합니다.
```c
int num = atoi("123"); // num 변수에 정수 123이 저장됨
```

### 3. itoa : 정수 → 문자열 변환 (Integer To Ascii)

- 기능: 정수(int)를 화면에 출력하거나 전송하기 위해 문자열(문자 배열)로 변환합니다.
- 헤더: <stdlib.h>
- 주의사항: 비표준 함수입니다. Windows(Visual Studio) 환경에서는 잘 작동하지만 Linux 등에서는 지원하지 않을 수 있습니다.
```c
char buf[10];
_itoa(123, buf, 10); // 정수 123을 10진수 문자열로 변환하여 buf에 "123" 저장
```

# 📌 C언어 문자열 결합: strcat vs sprintf

두 함수 모두 여러 문자열을 하나로 합칠 때 사용하지만, 작동 방식과 활용 목적이 완전히 다릅니다.

## 1. strcat (String Concatenate)

- 개념: 기존 문자열의 '맨 끝'을 찾아가서 새로운 문자열을 이어 붙이는 함수입니다.
- 헤더: <string.h>
- 특징: 대상 버퍼가 반드시 \0(Null 문자)로 초기화되어 있어야 안전하게 작동합니다.
### 💻 기본 문법 및 예제

```c
char buff[40] = "Hello "; // 반드시 여유 공간이 있어야 함
char str[] = "World";

strcat(buff, str); // buff 뒤에 str을 이어 붙임
// 결과: buff ==> "Hello World"
```

## 2. sprintf (String Print Formatted)

- 개념: printf가 모니터 화면에 출력하듯, 그 출력 내용을 문자열 변수(버퍼)에 그대로 써넣는(저장하는) 함수입니다.
- 헤더: <stdio.h>
