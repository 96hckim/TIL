## 📞 1. 폴링 (Polling)이란?

- 개념: CPU가 주기적으로 대상을 직접 찾아가서 "너 무슨 일 있어?", "버튼 눌렸어?" 하고 상태를 계속 확인하는 방식입니다.
- 비유 (택배 기다리기): 택배가 왔는지 궁금해서 5분마다 직접 현관문을 열고 나가서 확인하는 것과 같습니다.
- 우리가 짠 코드 예시: while(1) 루프 안에서 get_button_state() 함수를 무한 반복하며 버튼이 눌렸는지 계속 감사하던 방식이 바로 폴링입니다.
- 장점: 구현하기가 매우 쉽고 직관적입니다.
- 단점: 상태가 변하지 않았는데도 계속 확인하러 가야 하므로 CPU의 불필요한 연산 낭비(낭비되는 전력과 시간)가 심합니다. 만약 주기(_delay_ms(1))가 너무 길면 그 사이에 일어난 순간적인 입력을 놓칠 수도 있습니다.
## 🔔 2. 인터럽트 (Interrupt)란?

- 개념: CPU는 자기 할 일(예: FND 화면 그리기)을 열심히 하고 있다가, 외부 장치에 사건(Event)이 발생하면 CPU에게 신호("딩동!")를 보내 하던 일을 잠시 멈추고 그 사건을 처리하게 만드는 방식입니다.
- 비유 (택배 기다리기): 현관문에 초인종을 달아두고, 나는 방에서 내 할 일을 하다가 초인종이 울리는 순간에만 나가서 택배를 받는 것과 같습니다.
- 작동 방식: 외부에서 신호가 들어오면 CPU는 하던 일을 멈추고 ISR(Interrupt Service Routine, 인터럽트 서비스 루틴)이라는 특별한 함수로 점프하여 일을 처리한 뒤, 다시 원래 하던 일로 돌아옵니다.
- 장점: 이벤트가 발생했을 때만 CPU가 움직이므로 에너지와 자원 효율성이 극대화됩니다. 신호가 들어오는 즉시 반응하므로 반응 속도가 매우 빠릅니다.
- 단점: 인터럽트 핀 설정, 레지스터 설정 등 하드웨어 초기화 코드가 복잡하고, 여러 인터럽트가 동시에 터질 때의 우선순위를 고려해야 하므로 설계가 까다롭습니다.
## 📊 3. 폴링 vs 인터럽트 핵심 비교 요약

| 구분 | 🔄 폴링 (Polling) | 🔔 인터럽트 (Interrupt) |
|---|---|---|
| 주체 | CPU가 주도하여 상태를 확인 | 외부 장치가 주도하여 CPU에게 알림 |
| CPU 효율성 | 낮음 (무한 루프 돌며 자원 낭비) | 높음 (필요할 때만 동작) |
| 반응 속도 | 주기 체크 타이밍에 따라 지연 가능 | 즉각 반응 (실시간성 우수) |
| 구현 난이도 | 매우 쉬움 | 복잡함 (하드웨어 레지스터 제어 필요) |
| 적합한 상황 | 확인 주기가 자유롭고 단순한 시스템 | 중요도가 높고 언제 터질지 모르는 비정기적 사건 |

## 🏎️ 1. 컴파일러의 똑똑한(?) 짓: 최적화(Optimization)

C언어 컴파일러는 코드를 기계어로 바꿀 때, 최대한 속도를 빠르게 하기 위해 코드를 제멋대로 압축하거나 효율적으로 다듬는 '최적화'를 수행합니다.

예를 들어, 하드웨어 레지스터 포트(FND_DATA_PORT)에 값을 연속으로 넣는 코드를 짰다고 해봅시다.

```c
/* volatile이 없는 일반적인 코드 */
int* fnd_port = (int*)0x0025; // 임의의 FND 포트 주소

*fnd_port = 1;
*fnd_port = 2;
*fnd_port = 3;
```

우리의 의도는 FND에 1을 켰다가, 2를 켰다가, 최종적으로 3을 켜는 것입니다. 하지만 똑똑한 컴파일러가 이 코드를 보면 다음과 같이 생각합니다.

> "어차피 똑같은 주소에 1, 2 넣었다가 바로 3을 넣네? 앞의 1, 2는 넣으나 마나 최종 결과는 3이잖아? 중간 과정은 시간 낭비니까 삭제하자!"

결국 컴파일러는 앞의 두 줄을 청소해 버리고 *fnd_port = 3; 단 한 줄만 실행 파일로 만듭니다. 컴퓨터 화면에 띄우는 프로그램이라면 이게 맞을지 몰라도, 하드웨어를 제어하는 임베디드에서는 1과 2가 켜지는 과정이 통째로 생략되는 대참사가 발생합니다.

## 🛡️ 2. volatile이 해결하는 방법

이때 변수 선언 앞에 volatile을 붙여주면 컴파일러의 참견을 막을 수 있습니다.

```c
/* volatile을 붙인 안전한 코드 */
volatile int* fnd_port = (int*)0x0025;

*fnd_port = 1; // 삭제 안 됨! 그대로 실행
*fnd_port = 2; // 삭제 안 됨! 그대로 실행
*fnd_port = 3; // 그대로 실행
```

volatile이 붙은 변수는 컴파일러가 "아, 이 주소는 내가 모르는 외부 하드웨어나 무언가에 의해 실시간으로 값이 쓰이고 읽히는 곳이구나"라고 판단하여, 코드의 단 한 줄도 생략하지 않고 메모리(또는 레지스터)에 있는 실제 값을 매번 정직하게 읽고 씁니다.

## 🚀 3. 임베디드 실무에서 volatile을 무조건 써야 하는 3가지 상황

실무나 학업에서 volatile을 빼먹으면 "코드는 분명히 맞는데 하드웨어가 먹통이 되는" 미치고 팔짝 뛸 버그가 발생합니다. 주로 아래 3가지 상황에서 필수적으로 사용됩니다.

### ① 인터럽트 서비스 루틴(ISR)에서 공유하는 전역 변수

우리가 방금 공부한 인터럽트 함수와 메인 루프가 동시에 공유하는 변수에는 무조건 붙여야 합니다.

```c
volatile int g_button_pressed = 0; // 메인과 ISR이 공유하므로 volatile 필수!

int main(void){
    while(g_button_pressed == 0) {
        // g_button_pressed가 1이 될 때까지 무한 대기
        // 💡 volatile이 없으면 컴파일러가 "루프 내에서 이 변수를 바꾸는 코드가 없네?" 하고
        // 조건을 상수로 고정해버려, 인터럽트가 터져서 값이 바뀌어도 루프를 탈출하지 못합니다!
    }
}

ISR(BUTTON_interrupt) {
    g_button_pressed = 1; // 버튼이 눌리면 1로 변경
}
```

### ② 메모리 맵 입출력(MMIO) 하드웨어 레지스터

AVR의 PORTB, DDRC 같은 레지스터나 센서 데이터 주소값들은 CPU 외부에 있는 하드웨어 시그널에 의해 값이 수시로 변하므로, 무조건 volatile로 선언되어 있습니다. (실제로 avr/io.h 헤더 파일을 열어보면 모든 포트가 volatile로 매크로 정의되어 있습니다.)

### ③ 딜레이(Delay) 기능을 위한 루프 카운트 변수

아무 일도 안 하고 시간만 끌기 위해 무의미한 빈 반복문을 돌릴 때가 있습니다.

```c
volatile int delay_count;
for(delay_count = 0; delay_count < 10000; delay_count++);
// 💡 volatile이 없으면 컴파일러가 "아무 기능도 안 하는 쓸데없는 반복문이네?" 하고
// 이 for문을 통째로 지워버려서 딜레이가 작동하지 않습니다.
```

## 📌 요약

- volatile이란? ➡️ 컴파일러에게 "이 변수는 내 허락 없이 멋대로 최적화하거나 생략하지 마라"고 명령하는 키워드.
- 왜 쓰나요? ➡️ 하드웨어 레지스터나 인터럽트 공유 변수처럼 메모리의 실제 값을 실시간으로 정직하게 읽고 써야 할 때 프로그램이 씹히는 현상을 방지하기 위해 사용합니다.


임베디드 시스템 아키텍처에서 "ISR(Interrupt Service Routine)은 최대한 짧고 빠르게 끝내야 한다"는 것은 절대 깨져서는 안 되는 가장 중요한 철칙 중 하나입니다.

인터럽트가 발생하면 CPU는 하던 일을 멈추고 초비상 모드로 ISR 함수를 실행하게 되는데, 이 함수가 길어지면 시스템 전체가 마비되는 도미노 현상이 발생합니다. 그렇게 되는 결정적인 이유 4가지를 알기 쉽게 정리해 드릴게요.

## 🛑 1. 다른 인터럽트가 씹히는 '지연(Latency)' 문제

기본적으로 CPU가 어떤 인터럽트를 처리하는 동안에는 다른 인터럽트가 들어오지 못하도록 문을 걸어 잠급니다(Global Interrupt Disable). 만약 버튼 인터럽트(ISR)를 처리하느라 0.1초 동안 머물러 있다면, 그 0.1초 사이에 들어온 센서 데이터나 통신 데이터 신호는 CPU가 인지하지 못하고 통째로 증발(Data Loss)하게 됩니다. ISR을 짧게 짜야 다른 비상 신호들을 놓치지 않고 실시간으로 받을 수 있습니다.

## ⏳ 2. 메인 루프(Main Loop)의 먹통 현상

우리가 구현했던 FND 디스플레이를 떠올려 보세요. 메인 루프(while(1))는 1ms마다 FND 자릿수를 바꾸며 쉴 새 없이 화면을 그려내고 있었습니다.

그런데 버튼 인터럽트 함수가 실행되어 CPU를 0.5초 동안 붙잡고 있으면, 메인 루프는 0.5초 동안 완전히 멈춰 서게 됩니다. 결과적으로 FND 화면이 깜빡이거나 멈추는 디스플레이 왜곡 현상이 눈으로 보일 만큼 심하게 발생합니다.

## 💥 3. 시스템 크래시(Stack Overflow) 위험

인터럽트가 실행되면 CPU는 복귀 주소와 현재 레지스터들의 상태를 메모리의 '스택(Stack)'이라는 임시 공간에 저장합니다.

인터럽트 내부 코드가 길어지거나 그 안에서 또 다른 함수를 복잡하게 호출하게 되면 스택 메모리를 과도하게 사용하게 됩니다. 재수가 없어서 스택 공간이 넘쳐나면(Stack Overflow) 프로그램이 완전히 뻗어버리는(Crash) 최악의 버그가 발생합니다.

## 🚫 4. ISR 내부에서 절대 쓰면 안 되는 금기 사항들

코드가 길어지는 주범이자, ISR을 파괴하는 대표적인 금기 요수 2가지가 있습니다.

- _delay_ms() 같은 대기 함수: ISR 안에서 시간을 끌기 위해 딜레이 함수를 넣는 것은 CPU에게 "아무것도 하지 말고 비상 모드 상태로 대기해!"라고 하는 것과 같습니다. 절대 금물입니다.
- printf() 같은 무거운 I/O 함수: printf는 수많은 메모리 연산과 느린 시리얼 통신을 사용하는 엄청나게 무거운 함수입니다. ISR 안에서 확인용으로 printf를 남발하면 시스템이 수밀리초 동안 멈추게 됩니다.
## 💡 정석적인 해결책: "깃발(Flag)만 올리고 튀어라!"

그렇다면 인터럽트가 발생했을 때 처리해야 할 일이 많다면 어떻게 해야 할까요? 실무에서는 "ISR에서는 최소한의 기록만 하고, 실제 무거운 연산은 메인 루프에게 떠넘기는 방식"을 사용합니다.

```c
/* 1. 공유 변수는 컴파일러 최적화를 막기 위해 volatile 선언 */
volatile uint8_t g_button_flag = 0;

/* 2. 인터럽트 함수는 초고속으로 끝냅니다. */
ISR(INT0_vect)
{
    g_button_flag = 1; // "나 버튼 눌렸어!" 하고 깃발(Flag)만 올리고 0.00001초 만에 탈출! ✅
}

/* 3. 무거운 실제 처리는 메인 루프가 한가할 때 처리합니다. */
int main(void){
    // 초기화...
    while(1)
    {
        Fnd_Display(); // 배경에서 FND는 계속 부드럽게 돌아감

        // 인터럽트가 올려둔 깃발이 발견되면 그때 서야 무거운 작업을 처리
        if (g_button_flag == 1)
        {
            _delay_ms(50); // 디바운싱 처리 등 무거운 일은 여기서 안전하게!
            printf("버튼이 처리되었습니다.\n");

            g_button_flag = 0; // 처리 완료 후 깃발 내리기
        }
    }
}
```

이렇게 짜야 인터럽트의 본질인 "빠른 실시간 반응성"과 메인 루프의 "안정적인 하드웨어 구동"이라는 두 마리 토끼를 모두 잡을 수 있습니다!



## 🕒 1. 타이머(Timer)와 카운터(Counter)의 본질

하드웨어 칩 내부에는 아주 정확하게 일정한 간격으로 똑딱거리는 '시계 신호(클록, Clock)'가 들어옵니다.

- 타이머: 이 내부 클록 신호를 받아서 "하나, 둘, 셋..." 숫자를 세는 장치입니다. 일정한 속도로 숫자를 세기 때문에, 숫자가 올라간 양을 보면 정확히 시간이 얼마나 흘렀는지 알 수 있습니다.
- 카운터: 내부 시계 대신, 외부 핀으로 들어오는 신호(예: 버튼이 눌리는 횟수)를 세는 장치입니다.
## 💾 2. 왜 하필 '8비트' 타이머일까?

'8비트'라는 말은 숫자를 저장하는 방(레지스터)의 크기가 8비트라는 뜻입니다. 8비트로 표현할 수 있는 숫자의 범위는 이진수로 0000 0000부터 1111 1111까지, 즉 십진수로 0부터 255까지입니다.

8비트 타이머는 다음과 같이 작동합니다:

1. 클록 신호가 한 번 올 때마다 숫자가 0 ➡️ 1 ➡️ 2 ➡️ ... ➡️ 254 ➡️ 255까지 1씩 올라갑니다.
1. 255까지 꽉 찬 상태에서 클록이 한 번 더 오면, 숫자가 다시 0으로 리셋됩니다.
1. 이 자리가 넘쳐서 0으로 돌아가는 순간을 오버플로우(Overflow)라고 부르며, 이때 "타이머 오버플로우 인터럽트"라는 비상 벨이 울립니다.
## ⏳ 3. 시간이 너무 빨리 가는데? 프리스케일러(Prescaler)의 마법

만약 아두이노(ATmega328)가 16MHz 클록으로 작동한다면, 1초에 무려 16,000,000번 똑딱거립니다.

이 속도 그대로 8비트 타이머가 숫자를 세면 255까지 채우는 데 고작 0.000016초밖에 걸리지 않습니다. 사람이 인지하기엔 너무 순식간이죠.

그래서 시계 신호를 중간에 뻥튀기(생략)해 주는 프리스케일러(Prescaler)라는 중계기를 둡니다.

- 분주비(Prescaler Ratio) 선택: 예를 들어 분주비를 64로 설정하면, 외부에서 클록이 64번 들어올 때마다 타이머 숫자를 딱 1씩 올립니다.
- 효과: 시계 바늘이 64배 느리게 가도록 조절하는 것입니다. 이 분주비를 잘 계산하면 우리가 원하는 정확한 시간(예: 딱 1ms 또는 10ms)마다 인터럽트 비상벨이 울리도록 세팅할 수 있습니다.
## 🛠️ 4. 8비트 타이머의 3가지 핵심 모드

타이머는 단순히 시간만 재는 게 아니라 레지스터 설정에 따라 다양한 모드로 변신합니다.

### ① 일반 모드 (Normal Mode)

방금 설명해 드린 대로 0부터 255까지 정직하게 세다가 오버플로우가 나면 인터럽트를 발생시키는 가장 기본 모드입니다.

### ② CTC 모드 (Clear Timer on Compare Match)

255까지 가기 전에, 내가 원하는 목표치(예: 120)를 미리 적어두는 모드입니다. 타이머가 숫자를 세다가 120이 되는 순간 "목표 달성!" 인터럽트를 울리고 그 즉시 숫자를 0으로 리셋합니다. 주기적인 시간을 아주 정확하게 만들 때 씁니다.

### ③ PWM 모드 (Pulse Width Modulation)

FND 제어할 때는 잘 안 쓰지만, 모터 속도를 조절하거나 LED 밝기를 은은하게 조절할 때 쓰는 모드입니다. 타이머가 숫자를 올리는 도중에 특정 값보다 낮으면 전기를 켜고, 높으면 전기를 끄는 방식으로 고속의 사각파 신호를 만들어냅니다.

## 📌 요약: 임베디드 과제에서 타이머를 쓰는 이유

우리가 앞서 배웠던 _delay_ms() 함수는 시간 동안 CPU가 아무것도 안 하고 멍하니 멈춰 서 있는 '나쁜 대기' 방식입니다.

반면 타이머 인터럽트를 사용하면, 메인 루프는 FND를 켜거나 센서 값을 읽는 등 자기 할 일을 바쁘게 하고 있다가, 타이머가 정확히 1ms마다 초인종을 울려줄 때만 아주 잠깐 실행되므로 디스플레이가 끊기거나 버튼 입력이 씹히는 현상을 완벽하게 막을 수 있습니다.



## 💾 1. 8비트 타이머 vs 16비트 타이머 핵심 비교

가장 큰 차이는 숫자를 담을 수 있는 '바구니(레지스터)의 크기'와 그로 인해 발생하는 '최대 측정 시간'입니다.

| 구분 | ⏳ 8비트 타이머 (Timer 0, 2) | ⏳ 16비트 타이머 (Timer 1) |
|---|---|---|
| 카운터 범위 | 0 ~ 255 (2^8) | 0 ~ 65,535 (2^16) |
| 해상도 (Resolution) | 낮음 (정밀도가 비교적 떨어짐) | 높음 (매우 촘촘하고 세밀하게 제어 가능) |
| 주요 용도 | 주기적인 1ms 인터럽트, LED 밝기 조절 | 정밀한 서보모터 제어, 몇 초 단위의 긴 시간 측정 |
| 오버플로우 주기 | 매우 짧음 (클록이 몇 번만 튀어도 리셋) | 매우 길음 (숫자를 한참 동안 셀 수 있음) |

- 쉽게 이해하는 비유: 8비트 타이머는 "종이컵"이고 16비트 타이머는 "큰 양동이"입니다. 물(클록)을 채울 때 종이컵은 금방 넘쳐서(Overflow) 자주 비워줘야(인터럽트 발생) 하지만, 양동이는 오랜 시간 동안 물을 가만히 담아둘 수 있습니다.
## ⚙️ 2. 타이머를 지배하는 3대 핵심 레지스터

코드를 짤 때 타이머를 세팅하기 위해 건드리는 레지스터들은 이름 뒤에 타이머 번호(0, 1, 2)가 붙습니다. (예: Timer 1 이라면 TCCR1A, TCNT1 등)

1. TCCRn (Timer/Counter Control Register): 타이머의 '설정 변속기'입니다. 분주비(Prescaler)를 몇으로 할지, CTC 모드로 돌릴지 일반 모드로 돌릴지 등 모든 작동 방식을 여기서 결정합니다.
1. TCNTn (Timer/Counter Register): 숫자가 실제로 0, 1, 2... 올라가는 '계기판'입니다. 8비트 타이머는 이 계기판이 255를 넘으면 터지고, 16비트 타이머는 65535를 넘으면 터집니다.
1. OCRn (Output Compare Register): CTC 모드에서 사용하는 '목표치 스티커'입니다. 만약 OCR1A = 20000;이라고 적어두면, 계기판(TCNT1)이 올라가다가 20000이 되는 순간 딱 맞춰 인터럽트가 터집니다.
## 🔄 3. 타이머/카운터 전체 동작 흐름 모식도

타이머가 내부에서 어떻게 신호를 가공해서 인터럽트를 뿜어내는지 일직선으로 요약하면 다음과 같습니다.

```plain text
[1] 메인 시스템 클록 (예: 16MHz)
       ⬇️
[2] 프리스케일러 (Prescaler) ➡️ 분주비(1, 8, 64, 256, 1024)를 적용해 속도를 늦춤
       ⬇️
[3] TCNTn 레지스터 ➡️ 늦춰진 속도로 숫자를 째깍째깍 올림 (8비트는 255까지, 16비트는 65535까지)
       ⬇️
[4] 조건 충족 (오버플로우가 나거나, OCRn에 설정한 목표값에 도달하거나)
       ⬇️
[5] 인터럽트 발생 ➡️ CPU가 하던 일을 멈추고 ISR() 함수를 실행!
```
